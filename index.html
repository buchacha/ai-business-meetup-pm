NotebookLM

Погрузиться в требования заказчика за 15 минут

Загружаем расшифровку встречи с заказчиком или демо-файл, задаём вопросы и получаем структурированный BRD. NotebookLM отвечает строго по источникам — не галлюцинирует.

Промпт для NotebookLM
На основе загруженных материалов встречи:
1. Выдели ключевые требования заказчика (топ-7)
2. Определи бизнес-цели, которые он хочет достичь
3. Найди противоречия или неясности в требованиях
4. Составь черновик BRD со структурой:
   - Контекст и цель
   - Целевая аудитория
   - Ключевые требования
   - Открытые вопросы к заказчику

DeepSeek

Из требований заказчика — в User Stories и PRD

Берём BRD из кейса 1, структурируем User Stories и генерируем полный PRD для всей команды. DeepSeek отлично работает с длинными документами и доступен без VPN.

Промпт 1 — User Stories
Ты — senior product manager. На основе следующего BRD создай
User Story Map для MVP:
[ВСТАВИТЬ BRD ИЗ КЕЙСА 1]

Структура:
- Разбей на 3–5 эпиков (крупные функциональные блоки)
- Для каждого эпика: 3–5 историй формата:
  "Как [роль], я хочу [действие], чтобы [ценность]"
- Для каждой истории: 2–3 Acceptance Criteria
- Приоритет: Must Have / Should Have / Nice to Have
Формат: структурированный Markdown.
Промпт 2 — Полный PRD
На основе BRD и User Stories выше напиши полный PRD для команды.
PRD должен содержать:
1. Executive Summary (3–5 предложений)
2. Problem Statement
3. Goals & Success Metrics (OKR-формат)
4. User Personas (2–3 персоны)
5. Feature List с приоритетами MoSCoW
6. User Flows (текстовое описание ключевых сценариев)
7. Out of Scope — что НЕ делаем в MVP
8. Технические требования верхнего уровня
9. Таймлайн / Роадмап (квартальный)
10. Риски и зависимости
Язык: профессиональный, без воды. Формат: Markdown.

Claude — Главный кейс

ТЗ, Бэклог и Роадмап из PRD

Claude — лучший инструмент для длинных технических документов. Понимает контекст, не теряет его в диалоге, идеален для итеративной доработки.

Промпт А — Техническое задание
Ты — опытный технический аналитик. На основе следующего PRD
разработай Техническое задание (ТЗ) на разработку продукта.
[ВСТАВИТЬ PRD]

ТЗ должно включать:
1. Общее описание системы и архитектурный подход
2. Функциональные требования (детализация каждой фичи)
3. Нефункциональные требования (производительность, безопасность, масштабируемость)
4. Описание API и интеграций
5. Модель данных (сущности и связи)
6. Требования к UI/UX (ключевые экраны)
7. Критерии приёмки для каждого модуля
8. Глоссарий терминов
Формат: Markdown, структура ГОСТ-совместимая.
Промпт Б — Продуктовый Бэклог
На основе ТЗ выше создай продуктовый бэклог для первых двух спринтов
(2 недели каждый, команда 3–5 человек):

Для каждой задачи:
- ID (EPIC-XXX)
- Название (до 10 слов)
- Тип: Feature / Tech / Research / Bug
- Story Points (шкала Фибоначчи: 1, 2, 3, 5, 8, 13)
- Зависимости
- Acceptance Criteria (2–3 пункта)
- Приоритет: Critical / High / Medium / Low

Организуй по эпикам. Итого: 15–20 задач для MVP.
Формат: Markdown-таблица.
Промпт В — Квартальный Роадмап
На основе бэклога выше составь квартальный роадмап:
- Q1: MVP (что войдёт в первый релиз)
- Q2: Итерация (что добавим после фидбека)
- Q3: Масштабирование (рост, новые фичи)

Для каждого квартала: список эпиков, ключевые метрики успеха,
зависимости и риски.
Формат: Markdown с таблицей и кратким нарративом.

Claude + Mermaid

Специфицировать логику продукта в диаграммы

Claude генерирует Mermaid-код, mermaid.live рендерит его в картинку. Описываем логику текстом — получаем готовые диаграммы для документации и презентации.

Промпт А — User Flow (диаграмма потока)
На основе следующего описания продукта создай диаграмму
пользовательского потока (User Flow) в формате Mermaid flowchart:
[ВСТАВИТЬ ОПИСАНИЕ КЛЮЧЕВОГО СЦЕНАРИЯ]

Покажи:
- Точку входа пользователя
- Все ключевые шаги и решения (if/else развилки)
- Успешный финал и ошибочные сценарии
- Взаимодействие с внешними системами (если есть)
Формат: Mermaid flowchart LR (слева направо).
Промпт Б — ER-диаграмма базы данных
Создай ER-диаграмму базы данных для следующего продукта
в формате Mermaid erDiagram:
[ВСТАВИТЬ ОПИСАНИЕ ПРОДУКТА ИЛИ ТЗ]

Включи:
- Все ключевые сущности (таблицы)
- Атрибуты каждой сущности (с типами данных)
- Связи между сущностями (один-ко-многим, многие-ко-многим)
- Primary и Foreign keys
Используй синтаксис Mermaid erDiagram.
Промпт В — Sequence Diagram (взаимодействие систем)
Создай диаграмму последовательности (Sequence Diagram)
в формате Mermaid для следующего сценария:
[ВСТАВИТЬ ОПИСАНИЕ ИНТЕГРАЦИИ ИЛИ API-ВЗАИМОДЕЙСТВИЯ]

Покажи:
- Участников (Пользователь, Frontend, Backend, БД, внешние API)
- Последовательность запросов и ответов
- Асинхронные операции (если есть)
- Обработку ошибок
Используй синтаксис Mermaid sequenceDiagram.

Собрать презентацию проекта для стейкхолдеров

Передаём Manus PRD и ТЗ — получаем готовую презентацию проекта с дизайном. Первая версия за 3 минуты, доработка под нужды — ещё 10.

Промпт для Manus
Создай презентацию продукта для стейкхолдеров на основе следующих материалов:
[ВСТАВИТЬ PRD и/или ТЗ]

Структура презентации:
1. Титульный слайд (название, краткое описание, дата)

5. Роадмап (квартальный, визуально)
6. Ключевые метрики успеха
7. Команда и ресурсы

Стиль: профессиональный, технологичный, тёмная тема.
Язык: русский.
Количество слайдов: 8–10.

Kimi

Проанализировать данные зарплат и визуализировать инсайты

Загружаем таблицу со статистическими данными по зарплатам → анализируем → получаем визуализацию с готовыми выводами для принятия HR и продуктовых решений.

Промпт для Kimi
Проанализируй загруженную таблицу с данными по зарплатам и выполни:

1. Общая статистика по датасету:
   - Диапазон зарплат (min, max, median, среднее)
   - Количество позиций и распределение по уровням (Junior/Middle/Senior)
   - Распределение по функциям / специализациям

2. Ключевые инсайты:
   - Топ-5 самых высокооплачиваемых специализаций
   - Разрыв в зарплатах между уровнями (в % и абсолютных значениях)
   - Аномалии и выбросы в данных

3. Визуализации:
   - Boxplot: зарплаты по уровням (Junior / Middle / Senior)
   - Bar chart: медианные зарплаты по специализациям
   - Heatmap: зарплаты по уровню и функции (если есть данные)

4. Выводы для принятия решений:
   - Где рыночная ставка выше внутренней вилки
   - Рекомендации по ревизии грейдов
   - Топ-3 риска с точки зрения удержания команды

Формат: структурированный отчёт с графиками и таблицей выводов.
